<p><b>BigTable</b> is a <a href="/wiki/Data_compression" title="Data compression">compressed</a>, high performance, and <a href="/wiki/Proprietary_software" title="Proprietary software">proprietary</a> <a href="/wiki/Database_management_system" title="Database management system">database system</a> built on <a href="/wiki/Google_File_System" title="Google File System">Google File System</a> (also known as the "GFS"), <a href="/wiki/Distributed_lock_manager#Google.27s_Chubby_lock_service" title="Distributed lock manager">Chubby Lock Service</a>, <a href="/wiki/SSTable" title="SSTable">SSTable</a> and a few other <a href="/wiki/Google" title="Google">Google</a> technologies; it is currently not distributed nor is it used outside of Google, although Google offers access to it as part of their <a href="/wiki/Google_App_Engine" title="Google App Engine">Google App Engine</a>.</p>

<h2><span class="editsection">[<a href="/w/index.php?title=BigTable&amp;action=edit&amp;section=1" title="Edit section: History">edit</a>]</span> <span class="mw-headline" id="History">History</span></h2>
<p>BigTable development began in 2004<sup id="cite_ref-1st-blog_0-0" class="reference"><a href="#cite_note-1st-blog-0"><span>[</span>1<span>]</span></a></sup> and is now used by a number of Google applications, such as <a href="/wiki/MapReduce" title="MapReduce">MapReduce</a>, which is often used for generating and modifying data stored in BigTable,<sup id="cite_ref-1" class="reference"><a href="#cite_note-1"><span>[</span>2<span>]</span></a></sup> <a href="/wiki/Google_Reader" title="Google Reader">Google Reader</a>,<sup id="cite_ref-google-reader_2-0" class="reference"><a href="#cite_note-google-reader-2"><span>[</span>3<span>]</span></a></sup> <a href="/wiki/Google_Maps" title="Google Maps">Google Maps</a>,<sup id="cite_ref-maps-orkut_3-0" class="reference"><a href="#cite_note-maps-orkut-3"><span>[</span>4<span>]</span></a></sup> <a href="/wiki/Google_Book_Search" title="Google Book Search" class="mw-redirect">Google Book Search</a>, "My Search History", <a href="/wiki/Google_Earth" title="Google Earth">Google Earth</a>, <a href="/wiki/Blogger.com" title="Blogger.com" class="mw-redirect">Blogger.com</a>, <a href="/wiki/Google_Code" title="Google Code">Google Code</a> hosting, <a href="/wiki/Orkut" title="Orkut">Orkut</a>,<sup id="cite_ref-maps-orkut_3-1" class="reference"><a href="#cite_note-maps-orkut-3"><span>[</span>4<span>]</span></a></sup> <a href="/wiki/YouTube" title="YouTube">YouTube</a>,<sup id="cite_ref-4" class="reference"><a href="#cite_note-4"><span>[</span>5<span>]</span></a></sup> and <a href="/wiki/Gmail" title="Gmail">Gmail</a>.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5"><span>[</span>6<span>]</span></a></sup> Google's reasons for developing its own database include scalability and better control of performance characteristics.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6"><span>[</span>7<span>]</span></a></sup></p>
<h2><span class="editsection">[<a href="/w/index.php?title=BigTable&amp;action=edit&amp;section=2" title="Edit section: Design">edit</a>]</span> <span class="mw-headline" id="Design">Design</span></h2>
<p>BigTable maps two arbitrary string values (row key and column key) and timestamp (hence three dimensional mapping) into an associated arbitrary byte array. It is not a relational database and can be better defined as a sparse, distributed multi-dimensional sorted map.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7"><span>[</span>8<span>]</span></a></sup> BigTable is designed to scale into the <a href="/wiki/Petabyte" title="Petabyte">petabyte</a> range across "hundreds or thousands of machines, and to make it easy to add more machines [to] the system and automatically start taking advantage of those resources without any reconfiguration".<sup id="cite_ref-o.27reilly_8-0" class="reference"><a href="#cite_note-o.27reilly-8"><span>[</span>9<span>]</span></a></sup></p>
<p>Each <a href="/wiki/Table_(database)" title="Table (database)">table</a> has multiple dimensions (one of which is a <a href="/wiki/Field_(computer_science)" title="Field (computer science)">field</a> for time, allowing for <a href="/wiki/Versioning" title="Versioning" class="mw-redirect">versioning</a> and <a href="/wiki/Garbage_collection_(computer_science)" title="Garbage collection (computer science)">garbage collection</a>). Tables are optimized for GFS by being split into multiple <i>tablets</i> - segments of the table as split along a row chosen such that the tablet will be ~200 <a href="/wiki/Megabyte" title="Megabyte">megabytes</a> in size. When sizes threaten to grow beyond a specified limit, the tablets are compressed using the algorithm BMDiff<sup id="cite_ref-algorithms1_9-0" class="reference"><a href="#cite_note-algorithms1-9"><span>[</span>10<span>]</span></a></sup> (referenced in <sup id="cite_ref-10" class="reference"><a href="#cite_note-10"><span>[</span>11<span>]</span></a></sup>) and the Zippy compression algorithm<sup id="cite_ref-algorithms2_11-0" class="reference"><a href="#cite_note-algorithms2-11"><span>[</span>12<span>]</span></a></sup> publicly known and open-sourced as <a href="/wiki/Snappy" title="Snappy">Snappy</a>,<sup id="cite_ref-12" class="reference"><a href="#cite_note-12"><span>[</span>13<span>]</span></a></sup> which is a less space-optimal variation of <a href="/wiki/LZ77" title="LZ77" class="mw-redirect">LZ77</a> but more efficient in terms of computing time. The locations in the GFS of tablets are recorded as database entries in multiple special tablets, which are called "META1" tablets. META1 tablets are found by querying the single "META0" tablet, which typically resides on a server of its own since it is often queried by clients as to the location of the "META1" tablet which itself has the answer to the question of where the actual data is located. Like GFS's master server, the META0 server is not generally a <a href="/wiki/Bottleneck" title="Bottleneck">bottleneck</a> since the processor time, bandwidth necessary to discover and transmit META1 locations is minimal and clients aggressively cache locations to minimize queries.</p>
<h2><span class="editsection">[<a href="/w/index.php?title=BigTable&amp;action=edit&amp;section=3" title="Edit section: Other similar software">edit</a>]</span> <span class="mw-headline" id="Other_similar_software">Other similar software</span></h2>
<ul>
<li><a href="/wiki/Apache_Cassandra" title="Apache Cassandra">Apache Cassandra</a> — brings together Dynamo's fully distributed design and BigTable's data model. Written in Java.</li>
<li><a href="/wiki/HBase" title="HBase">HBase</a> — Written in Java. Provides BigTable-like support on the <a href="/wiki/Hadoop" title="Hadoop" class="mw-redirect">Hadoop</a> Core.<sup id="cite_ref-13" class="reference"><a href="#cite_note-13"><span>[</span>14<span>]</span></a></sup></li>
<li><a href="/wiki/Hypertable" title="Hypertable">Hypertable</a> — Hypertable is designed to manage the storage and processing of information on a large cluster of commodity servers.<sup id="cite_ref-hypertable_14-0" class="reference"><a href="#cite_note-hypertable-14"><span>[</span>15<span>]</span></a></sup></li>
<li><a rel="nofollow" class="external text" href="http://github.com/bluefish/kdi">KDI</a> — <a href="/wiki/Kosmix" title="Kosmix">Kosmix</a> attempt to make a BigTable clone. Written in C++.</li>
<li><a href="/wiki/LevelDB" title="LevelDB">LevelDB</a> - Google's embedded key/value store that uses similar design concepts as the BigTable Tablet<sup id="cite_ref-15" class="reference"><a href="#cite_note-15"><span>[</span>16<span>]</span></a></sup></li>
</ul>
